# ТЗ (актуализация MVP-модели данных)

Дата актуализации: **27 марта 2026**.

Этот документ фиксирует упрощение доменной модели для MVP и заменяет устаревшие части ТЗ, где использовались отдельные таблицы связей и отдельная таблица ученических credentials.

## 1) Целевая схема MVP
Используются только следующие доменные таблицы:
- `adult`
- `adult_role`
- `organization`
- `student`

## 2) Продуктовые правила MVP
1. У каждого ученика ровно один ответственный взрослый.
2. Ответственный взрослый может быть родителем или преподавателем.
3. Отдельная таблица связи ученика со взрослыми (старый `student_guardian_link`) не используется.
4. Отдельная таблица credentials ученика (старый `student_credentials`) не используется.
5. Источник истины для аутентификации — Supabase Auth.

## 3) Детализация сущностей
### adult
Профиль взрослого пользователя приложения.
Связь с Supabase Auth: `adult.auth_user_id -> auth.users.id`.

### adult_role
Назначение ролей взрослому:
- `role`: `parent | teacher | org_owner | org_teacher`
- `scope_type`: `personal | organization`
- `scope_id`: ссылка на `organization.id` для organizational scope

### organization
Организация для корпоративного контура.
Владелец хранится в `organization.owner_adult_id`.

### student
Профиль ученика.
Ключевые поля MVP:
- `owner_adult_id` (обязательный FK на `adult.id`)
- `owner_kind` (`parent | teacher`)
- `created_by_adult_id`
- `auth_user_id` (опциональный FK на `auth.users.id` для ученического входа)

## 4) Аутентификация
- Взрослый и ученик могут иметь отдельные UX-потоки входа.
- Оба потока обязаны использовать Supabase Auth.
- В доменных таблицах не хранятся пароли/хэши паролей.

## 5) Ограничения MVP
- Модель «несколько взрослых на одного ученика» перенесена на пост-MVP.
- Для MVP хранится только один ответственный взрослый на ученика.
