# Модель lesson-content для ShiDao MVP

**Статус:** Draft v1 / Proposed architecture baseline  
**Дата:** 6 апреля 2026  
**Область применения:** доменная модель методологических уроков и runtime-уроков без внедрения production-кода

---

## 1) Зачем нужен этот документ

Этот документ фиксирует **единый source of truth** для структуры учебного контента до начала реализации, чтобы команда не разошлась в трактовках на уровне backend, frontend и будущих интеграций (расписание, thread, homework, история уроков).

Проблема, которую он решает:

- без согласованной модели есть риск сделать «страницу под каждый урок» и зашить контент в UI-компоненты;
- есть риск перепутать методологический шаблон урока и проведённый урок по расписанию;
- есть риск раннего переусложнения схемы (таблица на каждый тип блока) либо, наоборот, потери типизации (неструктурированный blob без контрактов);
- есть риск смешать teacher-only данные и проекции для parent/student.

Почему это нужно до реализации:

- это определяет доменные границы между `methodology lesson`, `scheduled lesson`, `lesson thread`, `homework`;
- это задаёт контракт между контентом (данные) и рендерингом (UI), что критично для MVP и масштабирования;
- это снижает стоимость дальнейших миграций и refactor в доменном ядре.

## 2) Продуктовые принципы

### 2.1 Methodology как ядро продукта

В MVP методологии предзагружены в систему и выступают каноническим источником структуры уроков.

### 2.2 Lesson как основная рабочая единица преподавателя

Для teacher центральный объект операционной работы — урок (подготовка, проведение, пост-урочная фиксация).

### 2.3 Два связанных shell-контура: methodology и runtime

В модели есть **два разных shell-контура**, которые нельзя смешивать:

- **Methodology lesson shell** — канонический стабильный контур урока внутри методологии;
- **Scheduled lesson runtime shell (runtime contour)** — контур конкретного проведения для class+date/time.

Оба контура связаны, но имеют разные цели и разные поля; вариативное «тело» урока (ordered blocks) наследуется от methodology baseline.

### 2.4 Component != Content

UI-компоненты только отображают данные. Контент уроков не хранится в коде компонентов.

### 2.5 Typed blocks вместо hardcoded lesson pages

Урок собирается из типизированных блоков. Не допускается отдельная захардкоженная страница под каждый урок.

### 2.6 Один тип блока, разные инстансы

Одинаковый тип (`video`, `vocabulary`, `activity`) в разных уроках — это разные инстансы с разным содержимым и контекстом.

### 2.7 Reusable assets отделяются при независимой ценности

Ресурсы (песни, видео, рабочие листы, медиа), которые переиспользуются или имеют отдельный жизненный цикл, хранятся как отдельные сущности и связываются с блоками.

### 2.8 Methodology lesson отделён от scheduled lesson

Шаблон урока в методологии и конкретный урок класса на дату — разные сущности с разными целями.

### 2.9 Не переоптимизировать схему слишком рано

Для MVP выбирается минимально достаточная, расширяемая модель без table-per-block-type и без преждевременной детализации всех edge-cases.

## 3) Глоссарий

- **Methodology** — предзагруженная учебная программа/подход, содержащая последовательность методологических уроков.
- **Methodology lesson** — канонический урок внутри методологии (шаблон содержания и педагогического сценария).
- **Methodology lesson shell** — стабильный канонический контур урока в методологии (идентичность и навигация).
- **Scheduled lesson runtime shell / runtime contour** — runtime-контур конкретного проведения урока (дата/формат/статус/операционные поля).
- **Lesson block** — тип блока сценария (например, `video`, `song`, `activity`).
- **Lesson block instance** — конкретный экземпляр блока в конкретном уроке с собственным payload и порядком.
- **Reusable asset** — отдельный переиспользуемый ресурс (медиа, worksheet, словарь, файл), на который могут ссылаться блоки.
- **Scheduled lesson** — запланированная/проводимая реализация урока для конкретного class и даты/времени.
- **Lesson thread** — связанный коммуникационный поток вокруг scheduled lesson (сообщения, комментарии, уточнения).
- **Homework** — домашнее задание, привязанное к scheduled lesson и/или отдельным блокам урока.
- **Teacher projection** — полное представление данных урока для преподавателя (включая внутренние заметки/метаданные).
- **Parent/student projection** — ограниченная проекция урока для родителя/ученика (без teacher-only полей).

## 4) Два shell-контура: что где хранится

### 4.1 Methodology lesson shell (канонический контур)

Methodology shell отвечает на вопрос: **«что это за урок в методологии»**. Здесь живут стабильные поля:

1. **Название урока**.
2. **Позиция в методологии** (module/unit/lesson index).
3. **Сводка ключевой лексики/фраз** (короткий digest, не полный сценарий).
4. **Ориентировочная длительность** (estimated duration).
5. **Сводка связанных медиа** (например, «1 видео, 1 песня, 2 worksheet»).
6. **Methodology-level статус готовности/публикации** при необходимости (например: draft, ready, archived).

### 4.2 Scheduled lesson runtime shell / runtime contour

Runtime shell отвечает на вопрос: **«как проходит конкретный урок класса в конкретное время»**. Здесь живут operational/runtime поля:

1. **Scheduled date/time**.
2. **Формат проведения** (online/offline).
3. **Meeting link / place** (для online/offline).
4. **Runtime status** (planned / in_progress / completed / cancelled).
5. **Краткие class-specific teacher notes** (если нужны для этого проведения).

### 4.3 Что не входит ни в один shell

- детальные шаги сценария;
- длинные инструкции и полные тексты реплик;
- содержимое активностей по шагам;
- вложения «как есть» без структурирования (это body/asset layer);
- технические детали UI-рендеринга.

## 5) Что входит в lesson body

Lesson body — это **упорядоченный сценарий** из типизированных блоков (`lesson block instances`).

Почему порядок критичен:

- педагогический эффект зависит от последовательности (разогрев → ввод → практика → закрепление → завершение);
- порядок влияет на темп урока и переходы между активностями;
- one-click reorder в будущем возможен только при явной ordinal-модели.

Почему инстансы всегда lesson-specific:

- даже при одинаковом типе блока конкретные материалы, инструкции, примеры, длительность и подсказки различаются;
- это позволяет удерживать вариативность без дублирования типов.

Почему контент — это данные, а не UI-код:

- один и тот же блок должен рендериться в web/mobile/print-представлениях;
- данные должны быть доступны для аналитики, экспорта, thread/homework интеграций;
- контент не должен зависеть от жизненного цикла конкретного React-компонента.

Почему модель поддерживает и сходство, и вариативность:

- сходство: единая типизация блоков и общий контракт;
- вариативность: payload каждого block instance и ссылки на разные assets.

### 5.1 Короткие pseudo-data примеры block instance payload

> Псевдо-данные ниже — для снятия неоднозначности контрактов; это не SQL и не production-код.

**Пример A — `video_segment`**

## 6) Каноническая таксономия блоков для v1

Ниже — практический набор для MVP (9 типов), достаточный для методологически-центричных уроков китайского.

### 6.1 `intro_framing`

- **Назначение:** ввод в тему и цель урока.
- **Педагогический смысл:** задаёт фокус и ожидания.
- **Обязательные поля:** `title`, `goal`, `teacher_script_short`.
- **Опциональные поля:** `warmup_question`, `timebox_minutes`.
- **Хранение:** обычно embedded.
- **Пример:** «Сегодня тема: приветствия в школе, цель — 3 базовые фразы + мини-диалог».

### 6.2 `video_segment`

- **Назначение:** просмотр/разбор видео-фрагмента.
- **Педагогический смысл:** восприятие живой речи и контекста.
- **Обязательные поля:** `prompt_before_watch`, `focus_points`.
- **Опциональные поля:** `clip_start`, `clip_end`, `questions_after_watch`.
- **Хранение:** обычно ссылка на reusable video asset + embedded инструкции.
- **Пример:** короткий диалог «你好 / 你叫什么名字?» с вопросами на понимание.

### 6.3 `song_segment`

- **Назначение:** работа с песней/чантом.
- **Педагогический смысл:** ритм, фонетика, запоминание.
- **Обязательные поля:** `activity_goal`, `teacher_actions`.
- **Опциональные поля:** `lyrics_excerpt`, `repeat_count`, `movement_hint`.
- **Хранение:** чаще ссылка на reusable song asset; локальные подсказки — embedded.
- **Пример:** песенка с повторением тонов и ключевых слов темы.

### 6.4 `vocabulary_focus`

- **Назначение:** целевая лексика/фразы урока.
- **Педагогический смысл:** управляемый ввод единиц языка.
- **Обязательные поля:** `items` (термин, pinyin/транскрипция, перевод/смысл), `practice_mode`.
- **Опциональные поля:** `mnemonics`, `common_mistakes`, `mini_drill`.
- **Хранение:** гибридно — embedded для lesson-specific набора; ссылка на reusable vocabulary set при переиспользовании.
- **Пример:** 6 слов по теме «класс и предметы», плюс 2 фразы.

### 6.5 `teacher_prompt_pattern`

- **Назначение:** шаблоны реплик преподавателя.
- **Педагогический смысл:** унифицирует instructional language и scaffold.
- **Обязательные поля:** `prompt_patterns`, `expected_student_responses`.
- **Опциональные поля:** `adaptations_by_level`, `fallback_ru`.
- **Хранение:** обычно embedded (часто урок-специфично).
- **Пример:** «Спроси соседа: …», «Повтори за мной: …».

### 6.6 `guided_activity`

- **Назначение:** структурированная активность (парная/групповая/индивидуальная).
- **Педагогический смысл:** перенос в практику.
- **Обязательные поля:** `activity_type`, `steps`, `success_criteria`.
- **Опциональные поля:** `grouping`, `timebox_minutes`, `differentiation_notes`.
- **Хранение:** чаще embedded; при частом повторе — ссылка на reusable activity template.
- **Пример:** ролевой мини-диалог «учитель–ученик» по карточкам.

### 6.7 `materials_prep`

- **Назначение:** список подготовки и материалов.
- **Педагогический смысл:** снижает операционные риски до начала занятия.
- **Обязательные поля:** `materials_checklist`.
- **Опциональные поля:** `print_count`, `room_setup_notes`.
- **Хранение:** обычно ссылки на reusable assets + embedded checklist.
- **Пример:** карточки, маркеры, ссылка на презентацию, аудио-файл.

### 6.8 `worksheet_task`

- **Назначение:** работа с workbook/worksheet.
- **Педагогический смысл:** закрепление и контроль понимания.
- **Обязательные поля:** `task_instruction`, `completion_mode`.
- **Опциональные поля:** `answer_key_hint`, `home_extension`.
- **Хранение:** обычно ссылка на reusable worksheet asset + локальные инструкции.
- **Пример:** лист «соедини иероглиф и pinyin», затем проверка в парах.

### 6.9 `wrap_up_closure`

- **Назначение:** завершение урока.
- **Педагогический смысл:** фиксация результата и мост к следующему шагу.
- **Обязательные поля:** `recap_points`, `exit_check`.
- **Опциональные поля:** `preview_next_lesson`, `teacher_reflection_prompt`.
- **Хранение:** обычно embedded.
- **Пример:** «Назови 2 новые фразы, которые ты используешь сегодня».

**Почему именно этот набор для v1:** он покрывает типовой контур методологического урока, не раздувая модель десятками узкоспециализированных типов.

## 7) Reusable assets vs embedded content

Критерий выбора: **есть ли у контента независимый жизненный цикл и ценность переиспользования**.

### 7.1 Когда хранить прямо в block instance (embedded)

Храним внутри блока, если контент:

- уникален для конкретного урока;
- короткий и контекстный (локальная инструкция, точечная реплика, конкретная подводка);
- редко переиспользуется вне этого урока;
- не требует самостоятельного управления версиями/правами/медиа-метаданными.

### 7.2 Когда выносить в reusable asset

Выносим в отдельную сущность, если контент:

- используется в нескольких уроках/методологиях;
- имеет собственные атрибуты (автор, источник, длительность, язык, лицензия);
- может обновляться независимо от конкретного урока;
- должен индексироваться/искаться как библиотечный ресурс.

### 7.3 Практические правила по типам

- **Songs:** обычно reusable asset (аудио, текст, темп, права); в блоке — педагогические инструкции под урок.
- **Videos:** reusable asset (url/file, длительность, язык, субтитры); в блоке — фокус просмотра, вопросы, таймкоды.
- **Vocabulary:**
  - embedded, если набор уникален для урока;
  - reusable vocabulary set, если повторяется по курсу.
- **Worksheets:** reusable asset (файл, версия, уровень); в блоке — как использовать на занятии.
- **Teacher instructions:** чаще embedded, потому что сильно контекстны.
- **Activity scenarios:**
  - embedded для одноразовой активности;
  - reusable template, если это стабильная методическая механика.
- **Media/attachments:** reusable, если файл может жить отдельно и повторно подключаться.

## 8) Концептуальные связи доменной модели (без SQL)

### 8.1 Базовые связи

1. **Methodology** содержит множество **methodology lessons**.
2. **Methodology lesson** содержит:
   - один `methodology lesson shell`;
   - упорядоченный список `lesson block instances`.
3. Каждый **lesson block instance**:
   - имеет `block_type`;
   - имеет `order` внутри lesson body;
   - хранит `content payload`;
   - может ссылаться на 0..N **reusable assets**.

### 8.2 Runtime-слой

4. **Class** имеет множество **scheduled lessons** по датам.
5. **Scheduled lesson** обычно ссылается на один **methodology lesson** как baseline.
6. Scheduled lesson хранит runtime-данные (фактический статус, teacher/runtime notes, итоги проведения, фактические материалы).
7. **MVP policy:** scheduled lesson наследует methodology lesson как baseline и **не поддерживает arbitrary per-block overrides** канонического content payload.
8. Любые «изменения по месту» в MVP фиксируются как runtime notes/outcome notes вокруг scheduled lesson, а не как редактирование канонических block instances.

### 8.3 Интеграции вокруг урока

9. **Lesson thread** привязан к scheduled lesson (коммуникация вокруг конкретного проведения).
10. **Homework** привязан к scheduled lesson и при необходимости к конкретным block instances.
11. **Attendance/история** в будущем также привязывается к scheduled lesson, а не к methodology lesson.

### 8.4 Проекции

12. Из полного teacher domain view строятся разные read-проекции:
   - teacher projection (полная);
   - parent/student projection (ограниченная, безопасная, понятная).

## 9) Workflow преподавателя

### 9.1 До урока

- teacher открывает scheduled lesson на конкретную дату/класс;
- получает baseline из methodology lesson (methodology shell + ordered body);
- дополняет только runtime shell (дата/формат/meeting/place/notes) и проверяет `materials_prep`, медиа, worksheet.

### 9.2 Во время урока

- двигается по ordered blocks как по сценарию baseline;
- отмечает прогресс и фиксирует наблюдения как runtime notes/outcome notes;
- использует linked assets и встроенные инструкции блока.

### 9.3 После урока

- фиксирует итоги (что прошли, что потребовало повторения);
- формирует/прикрепляет homework;
- запускает/обновляет коммуникацию в lesson thread.

### 9.4 Как кооперируются methodology lesson и scheduled lesson

- methodology lesson задаёт канонический методический каркас (source of truth контента);
- scheduled lesson добавляет контекст конкретного класса и времени через runtime shell;
- в MVP runtime-контур **не** делает per-block override канонического контента;
- изменения runtime не должны «ломать» базовую методологию без отдельного процесса.

## 10) Видимость для parent и student

### 10.1 Что показывать родителю/ученику

- название и тема урока;
- ключевая лексика/фразы (в безопасной проекции);
- фактически пройденные части (в агрегированном виде);
- домашнее задание и материалы для выполнения дома;
- при необходимости — selected media links.

### 10.2 Что остаётся teacher-only / methodology-only

- внутренние методические заметки и подсказки преподавателю;
- детальные сценарные пометки по управлению классом;
- внутренние статусные/операционные поля;
- черновые и промежуточные материалы методологии.

## 11) Рекомендуемое направление хранения

Для MVP рекомендуется:

1. **Единая модель хранения lesson blocks**, без отдельной таблицы на каждый тип блока.
2. Для каждого block instance хранить минимум:
   - `block_type`;
   - `order`;
   - `content payload` (структурированные данные по типу);
   - ссылки на reusable assets (где применимо).
3. **Отдельные reusable сущности** для ресурсов с переиспользованием/жизненным циклом.

Почему это предпочтительно:

- снижает стоимость добавления новых типов блоков;
- удерживает баланс типизации и гибкости;
- исключает захардкоженные «урок-страницы»;
- проще масштабируется на thread/homework/projections;
- соответствует текущему MVP-фокусу на методологиях и ролях `parent/teacher/student`.

---

## 12) Non-goals для v1

Этот документ **не** пытается решить сейчас:

- редактор методологий (methodology editor) для конечных пользователей;
- полноценный no-code конструктор блоков общего назначения;
- AI-генерацию уроков/сценариев;
- систему сверх-кастомизированных страниц «под каждый урок»;
- исчерпывающую таксономию блоков для всех будущих edge-cases;
- финальные аналитические модели прогресса/оценивания.
