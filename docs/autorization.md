# Авторизация и роли (MVP, март 2026)

## Принцип
Аутентификация и хранение паролей выполняются только через **Supabase Auth** (`auth.users`).

Доменные таблицы приложения не хранят `password_hash` и не дублируют логику входа.

## Акторы
- **Взрослый**: профиль в таблице `adult`, связанный с `auth.users` через `adult.auth_user_id`.
- **Ученик**: профиль в таблице `student`; при отдельном входе может быть связан с `auth.users` через `student.auth_user_id`.

## Роли взрослых
Роли хранятся в `adult_role`:
- `role`: `parent | teacher | org_owner | org_teacher`
- `scope_type`: `personal | organization`
- `scope_id`: организация для `organization`-скоупа, `NULL` для `personal`.

Один взрослый может иметь несколько ролей и работать в нескольких скоупах.

## MVP-правило владения учеником
`student` хранит единственного ответственного взрослого:
- `owner_adult_id` → `adult.id`
- `owner_kind` ∈ (`parent`, `teacher`)

Это заменяет старую multi-link модель. Отдельной таблицы связи ученик↔родитель в MVP нет.

## Что удалено
- `student_credentials` — удалена.
- `student_guardian_link` — удалена.

## Политика доступа (базово)
- Взрослый видит свой профиль `adult` и свои записи в `adult_role`.
- Взрослый видит ученика, если:
  - он ответственный (`student.owner_adult_id = current_adult_id()`), или
  - у него преподавательская роль в соответствующем скоупе.

## UI-поток входа
На UI можно оставлять разделение «Я взрослый / Я ученик», но обе ветки должны завершаться входом через Supabase Auth.
